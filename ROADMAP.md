## 二、详细阶段计划

**目标**：完成架构定型与技术验证。

| 周次 | 任务 | 产出 | 负责人 |

| W3-W4 | ClickHouse 压测 | 100万 EPS 验证报告 | 数据工程师 |
| W3-W4 | Qdrant 向量压测 | 1亿向量 50ms 验证 | 数据工程师 |
| W5-W6 | eBPF PoC | 内核采集 Demo | Rust 工程师 |
| W5-W6 | OTLP 接入 PoC | 多语言 SDK 验证 | 后端工程师 |
| W7-W8 | LLM 选型评估 | 模型对比报告 | AI 工程师 |
| W7-W8 | K8s 基础设施 | 开发/测试环境 | SRE |

**里程碑**：M2 末完成架构评审，技术风险关闭。

## 四、研发流程

### 4.1 敏捷开发

- **迭代周期**：2 周 Sprint
- **节奏**：周一计划 → 周五 Demo → 周末回顾
- **工具**：Jira + Confluence + GitHub + Slack

### 4.2 代码质量

| 环节 | 工具 | 标准 |
|------|------|------|
| 代码规范 | clang-format/gofmt/black | 100% 通过 |
| 静态分析 | clippy/golangci-lint/ruff | 0 Error |
| 单元测试 | cargo test/go test/pytest | 覆盖率 > 80% |
| 集成测试 | Docker Compose + 脚本 | 核心流程覆盖 |
| 性能测试 | Locust + 基准 | 每版本回归 |
| 安全扫描 | Trivy + cargo audit | 0 高危 |
| Code Review | GitHub PR + 2 人审批 | 强制 |

### 4.3 发布流程

```
开发 → PR → CI(测试/扫描) → Code Review → 合并 main
  ↓
main → 自动构建镜像 → 部署 Staging → 集成测试
  ↓
Staging 通过 → 灰度 1% → 观察 1h → 灰度 10% → 观察 4h
  ↓
灰度 50% → 观察 24h → 全量发布
  ↓
任意阶段异常 → 自动回滚
```

## 五、风险评估与应对

### 5.1 技术风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| ClickHouse 大规模运维复杂 | 中 | 高 | 提前培养 + 厂商支持 |
| LLM 幻觉导致误判 | 高 | 高 | 置信度 + 溯源 + 人工复核 |
| WebGPU 兼容性 | 中 | 中 | WebGL 降级 |
| eBPF 内核版本兼容 | 中 | 中 | 多版本测试 + BTF |
| 区块链性能瓶颈 | 低 | 中 | Merkle 批量 + 侧链 |

### 5.2 项目风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 进度延期 | 高 | 高 | MVP 范围严格管控 + 优先级裁剪 |
| 人才招聘困难 | 高 | 高 | 远程办公 + 校招培养 + 外包 |
| 需求蔓延 | 中 | 高 | 严格变更评审 + MVP 思维 |
| 技术债务 | 中 | 中 | 每迭代 20% 时间还债 |

### 5.3 风险监控

- **周报**：每周风险更新，红黄绿灯。
- **月度评审**：每月风险回顾，调整应对。
- **应急预算**：预留 15% 工时应对突发。
